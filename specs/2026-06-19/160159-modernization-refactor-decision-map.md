# Composer Piano Trainer Modernization Refactor Decision Map

## Context

Composer Piano Trainer is currently a static React application served from `docs/` by `server.js`. It uses React and ReactDOM from CDN UMD bundles, Babel Standalone for in-browser JSX transformation, and browser globals for cross-file dependencies.

The app is being revived after sitting for a while. The goal is to put it in good order before new feature work resumes. The user is not a developer and wants the project moved toward a modern, useful standard rather than preserving the old setup for its own sake.

Recent audit work found useful first-pass correctness fixes and created a separate audit note at `notes/refactor-audit-2026-06-19.md`. This spec supersedes that note as the planning artifact for the staged refactor.

## Goals

- Modernize the app architecture to a common, maintainable React project shape.
- Improve game correctness and make behavior easier to reason about.
- Keep the app deployable as a live public page.
- Use GitHub Pages as the hosting target for now.
- Target Chrome and Chromium-based browsers, especially because Web MIDI support is central to the app.
- Preserve the current working trainer experience while reducing fragility.
- Refactor in stages rather than rewriting the app in one risky pass.

## Non-Goals

- Do not prioritize visual polish in the first refactor phase.
- Do not prioritize performance tuning beyond what naturally improves from removing in-browser Babel and development CDN scripts.
- Do not redesign the product experience yet.
- Do not expand browser support beyond Chrome/Chromium for the initial modernization.
- Do not require backend infrastructure.

## Current Architecture

Relevant paths:

- `docs/index.html`
- `docs/app.js`
- `docs/chord-trainer.js`
- `docs/music-theory.js`
- `docs/sidebar.js`
- `docs/hooks/useTimer.js`
- `docs/hooks/useGameState.js`
- `docs/presets.js`
- `docs/midi-utils.js`
- `server.js`
- `tests/run-tests.js`

Current traits:

- Static app served from `docs/`.
- React runs via CDN scripts.
- JSX is transformed in the browser using Babel Standalone.
- Application modules communicate through `window` globals.
- Core trainer, sidebar, and music theory files are large and hard to isolate.
- Pure behavior testing has only recently been introduced.

## Key Problems To Solve

### Browser-Global Coupling

The app depends on script order and globals such as `MusicTheory`, `Presets`, `window.hooks`, `Sidebar`, and `ChordTrainer`. This makes refactors brittle and hides dependency errors until runtime.

Decision: move toward explicit JavaScript module imports.

### Mixed Game State Ownership

`ChordTrainer` owns local display and game-flow state while `useGameState` owns reducer-based game state. This overlap has already created correctness bugs around score, lives, timeout behavior, and summary transitions.

Decision: consolidate game-owned state into a single reducer/state machine layer where practical.

### Oversized Files

`chord-trainer.js`, `music-theory.js`, and `sidebar.js` are each large enough that behavior is hard to audit.

Decision: split by domain and UI responsibility after tests cover key behavior.

### Weak Test Surface

The new lightweight tests cover only a narrow slice. More behavior should be protected before structural migration.

Decision: add tests around game correctness before moving large pieces.

### Legacy Runtime Setup

In-browser Babel and development React are useful for demos but not ideal for ongoing development.

Decision: migrate to a modern React build tool. Vite + React is the preferred default because it is lightweight, common, and produces static assets suitable for GitHub Pages.

## Hosting Decision

Use GitHub Pages for now.

Preferred deployment model:

- Source code lives in a modern app structure.
- Build output is generated by a command such as `npm run build`.
- GitHub Actions deploys the built static output to GitHub Pages.

Open implementation choice:

- Deploy from a `dist/` artifact using GitHub Actions, preferred for Vite.
- Avoid keeping generated build output committed unless the repository constraints make Actions impractical.

## Browser And MIDI Support

Target:

- Chrome
- Edge
- Other Chromium browsers with compatible Web MIDI behavior

The app should clearly handle:

- Web MIDI unavailable.
- MIDI permission denied.
- No MIDI devices connected.
- Multiple MIDI inputs.
- "All inputs" selection.
- Device reconnect or refresh behavior where reasonable.

Because the target is Chrome/Chromium, Web MIDI compatibility can be treated as a core browser capability rather than a cross-browser abstraction problem.

## Refactor Strategy

### Stage 1: Correctness Baseline

Purpose: make current behavior safer before structural migration.

Important behavior areas:

- Scoring and multiplier math.
- Lives and game-over transitions.
- Timeout handling.
- Wrong-answer handling.
- Question count and session completion.
- Practice mode behavior.
- Progression mode behavior.
- Chord validation, including inversions and optional fifths.
- MIDI input selection normalization.

Preferred shape:

- Keep tests runnable without a browser where possible.
- Add browser-level smoke tests only once the project has a modern local dev server.

### Stage 2: Module Boundaries

Purpose: remove browser-global dependency flow without changing product behavior.

Candidate module splits:

- `src/domain/musicTheory/*`
- `src/domain/scoring/*`
- `src/domain/gameState/*`
- `src/domain/midi/*`
- `src/data/presets/*`
- `src/components/trainer/*`
- `src/components/sidebar/*`
- `src/components/keyboard/*`

Key decision:

- Extract pure logic first, then UI components.
- Keep the UI visually close to current behavior during this stage.

### Stage 3: Vite Migration

Purpose: move to a modern React project standard.

Preferred decisions:

- Use Vite with React.
- Use ES modules.
- Keep the project as a static frontend.
- Preserve existing CSS initially to reduce product risk.
- Configure base path for GitHub Pages.

Likely files:

- `src/main.jsx`
- `src/App.jsx`
- `src/components/...`
- `src/domain/...`
- `vite.config.js`
- `package.json`
- `.github/workflows/deploy.yml`

Question for implementation:

- Determine repository name before final GitHub Pages `base` configuration.

### Stage 4: State Model Consolidation

Purpose: make the game loop predictable.

The desired end state is a single game reducer/state machine that owns:

- Current state: idle, playing, feedback, summary, etc.
- Lives.
- Score.
- Streak.
- Multiplier.
- Question count.
- Attempts.
- Current chord or progression step identity.
- Summary transition.
- Processing lock or equivalent guard.

React component state should remain for:

- UI-only toggles.
- Modal state.
- Selected settings before a game starts.
- Display-only derived state where appropriate.

### Stage 5: Component Decomposition

Purpose: make app work pleasant and understandable.

Candidate components:

- `TrainerPage`
- `QuestionDisplay`
- `ProgressionDisplay`
- `TimerBar`
- `LivesDisplay`
- `ScoreDisplay`
- `SessionControls`
- `GameSummary`
- `PianoKeyboard`
- `Sidebar`
- `PresetSelector`
- `ChordTypeSelector`
- `ProgressionSelector`
- `MidiInputSelector`

Candidate hooks:

- `useMidiInputs`
- `useTrainerGame`
- `usePersistentSettings`
- `useTimer`

## Product Behavior Decisions

- Keep the app focused on chord training first.
- Continue supporting practice mode.
- Continue supporting progression mode.
- Continue showing the piano keyboard and failed chord notes on game over.
- Continue allowing the fifth to be optional for supported larger chords.
- Preserve current presets unless tests reveal incorrect theory.
- Prefer correctness and clarity over adding new chord content during the refactor.

## Data And Domain Rules

Important domain rules to preserve or clarify:

- Root-position mode requires the bass note to be the chord root.
- Free inversion mode validates pitch classes without requiring a specific bass note.
- Specific inversion mode validates the expected bass note.
- Optional fifth applies to seventh-or-larger chords where currently intended, not plain sixth chords unless deliberately changed.
- Practice mode should not behave like a scored timed session.
- Progression mode counts a progression as a unit where currently intended, but this should be reviewed and made explicit in tests.

## Testing Decisions

Use layered tests:

- Pure unit tests for music theory, scoring, MIDI normalization, and reducer transitions.
- Component tests only after the module/build migration makes them practical.
- Browser smoke tests for app boot and main flows after Vite migration.

Critical tests to add before major moves:

- Correct answer increments score using response-time points and multiplier.
- Wrong answer decrements lives once and ends the game at zero.
- Timeout decrements lives once and ends the game at zero.
- Practice mode does not time out.
- Progression advances correctly and completes sessions correctly.
- MIDI "All inputs" maps to the internal all-inputs state.
- Common chord/inversion validation cases.

## GitHub Pages Constraints

The future implementation should account for:

- Static asset paths under a repository base path.
- Audio files and SVG assets included in the build output.
- Client-only routing avoided or configured safely.
- Deploy workflow that runs tests before publishing.

Preferred deployment pipeline:

1. Install dependencies.
2. Run tests.
3. Build static site.
4. Upload build artifact.
5. Deploy to GitHub Pages.

## Risks

- Large structural changes could break subtle game behavior if tests are too thin.
- Music theory edge cases may encode past bug fixes in ad hoc conditionals.
- Progression mode currently has more state complexity than single-chord mode.
- Web MIDI behavior is hard to fully automate without real devices.
- GitHub Pages base path can break assets if configured late or incorrectly.

## Open Questions For Implementation

- What is the GitHub repository name and final Pages URL?
- Should generated GitHub Pages output be deployed by Actions only, or committed to a branch?
- Should the existing `docs/` folder remain as historical source during migration, or be replaced once Vite is ready?
- Which current presets are most important to protect with regression tests first?
- Are there known music theory cases the user already distrusts or wants checked?

## Recommended First Implementation Slice

Start with a correctness-first slice before the full Vite migration:

- Expand tests around reducer transitions, scoring, timeout, wrong-answer game-over, and chord validation.
- Extract pure scoring and MIDI helpers into explicit modules.
- Reduce the coupling between `ChordTrainer` and `useGameState`.

Then migrate to Vite once behavior is better pinned down.

This staged approach gives the project a modern destination without turning the first step into a risky rewrite.
